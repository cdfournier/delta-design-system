# Fieldsets

Fieldsets provide consistent structure for form inputs with labels, validation states, and helper messages. They adapt layout and spacing based on input type, ensuring accessible and predictable form patterns.

## Anatomy

A fieldset consists of:

- **Label or legend:** Descriptive text that identifies the input or group. Single inputs use a `<label>`; choice input groups use a `<legend>` inside a `<fieldset>`.
- **Legend label (optional):** Fine-print text alongside the legend for supplementary context such as `(Optional)`; applies to checkbox, radio, and switch groups.
- **Input:** The interactive control: text field, password field, select, checkbox, radio, or switch.
- **Message (optional):** Helper text or validation feedback below the input.

## Variants

Fieldsets vary by input type. The type determines anatomy, layout, and interaction behavior.

- **Text:** Single-line text entry with optional left and right icon slots.
- **Password:** Text input with a show/hide toggle button.
- **Select:** Dropdown menu with a chevron indicator.
- **Checkbox:** Group of independent selection controls under a shared legend.
- **Radio:** Group of mutually exclusive selection controls under a shared legend.
- **Switch:** Group of binary toggle controls under a shared legend.

## States

- **Default:** Initial unfocused state.
- **Focus:** 2px primary-colored outline with 2px offset.
- **Disabled:** Non-interactive, reduced opacity and `not-allowed` cursor.
- **Read-only:** Non-editable text/select value that remains readable and submittable.
- **Invalid:** Border switches to invalid state color and error message is linked via `aria-describedby`.
- **Valid:** Border switches to valid state color where positive confirmation is useful.
- **Checkbox values:** `unchecked`, `checked`, and `indeterminate`.
- **Password visibility:** Hidden/shown via toggle control with updated label and icon.
- **Select open/closed:** Closed state shows value; activated state expands options.

## Accessibility

### Keyboard navigation

All fieldset inputs must be fully keyboard accessible:

- **Tab:** Moves focus between controls in reading order.
- **Shift + Tab:** Moves focus backward.
- **Space:** Toggles checkboxes and switches; opens a closed select.
- **Enter:** Submits the form and activates inline controls (for example, show/hide password).
- **Arrow keys:** Navigate between radio buttons within a group.

### Screen readers

Use semantic HTML so assistive technology can parse form structure correctly. Use `<fieldset>` to group related controls and `<legend>` for group naming. For text/password/select, use explicit `for` and `id` associations.

### ARIA attributes

- **aria-invalid="true":** Applied to invalid inputs.
- **aria-required="true":** Marks required fields.
- **aria-describedby:** Links helper/error text to its input.
- **aria-disabled="true":** For controls where native `disabled` cannot be used.
- **role="switch":** Applied to switch-style checkbox controls.
- **role="alert":** Applied to error messages that should be announced immediately.

### Focus management

Interactive controls require a visible focus indicator. Keep logical tab order and do not suppress focus outlines.

### Color contrast

Label and input text satisfy WCAG AA thresholds in both light and dark modes. Placeholder text must never be the only source of required guidance.

## Usage

### When to use

Use Fieldsets when you need to:

- Collect user input with consistent labeling and structure.
- Group related checkbox/radio/switch controls under a shared label.
- Provide inline helper and validation feedback.
- Maintain accessible form semantics for keyboard and assistive tech users.

### When not to use

Do not use Fieldsets when:

- Building search inputs with dedicated search semantics.
- Creating filter/toolbar controls better represented by other patterns.
- You think a field does not need a visible label; use accessible labeling instead of removing labels.

### Content guidelines

- **Write concise labels:** 1-3 words that clearly describe the expected value.
- **Use placeholders for examples:** Use format hints, not duplicate labels.
- **Write proactive helper text:** Set expectations before errors occur.
- **Keep legends brief:** Group labels should add context without repetition.
- **Write actionable errors:** Explain what failed and how to fix it.

## Code reference

### HTML

```html
<!-- Text input fieldset -->
<fieldset class="fieldset">
  <label for="email">Email address</label>
  <input type="email" id="email" placeholder="[email protected]" aria-describedby="email-message" />
  <div id="email-message" class="message">We'll never share your email with anyone else.</div>
</fieldset>

<!-- Checkbox group fieldset -->
<fieldset class="fieldset fieldset--choice">
  <legend>Interests <span class="legend-label">(Optional)</span></legend>
  <label><input type="checkbox" value="design" /> Design</label>
  <label><input type="checkbox" value="development" /> Development</label>
  <label><input type="checkbox" value="marketing" /> Marketing</label>
  <div class="message">Select all that apply.</div>
</fieldset>
```

### CSS

```css
.fieldset {

  border: none;
  padding: 0;

  gap: 0.5rem;

  font-size: 0.75rem;
  line-height: 1.25rem;
  font-weight: 700;

  font-size: 0.625rem;
  line-height: 1.125rem;
}
```

### JavaScript

```js
export function createFieldset({ type = 'text', state = 'default', showMessage = true, legendLabel = false } = {}) {
  if (type === 'checkbox' || type === 'radio' || type === 'switch') {
    return choiceFieldset({ type, state, showMessage, legendLabel });
  }

  return textLikeFieldset({ type, state, showMessage });
}
```
