# Tabs

Tabs organize related content into separate panels, allowing users to switch between views without leaving the page. The Tabs system is a compound component consisting of tab buttons, a tablist container, content panels, and the assembled tab interface.

## Anatomy

The Tabs component system consists of four sub-components:

- **Tab button:** Individual trigger button for switching panels.
- **Tablist:** Container holding all tab buttons.
- **Tab panel:** Content area displayed for the active tab, containing an optional figure and a heading with body text (see the Content component for typography details).
- **Tab (assembled):** Complete system combining a tablist and tab panels.

## Variants

- **Horizontal:** Tablist appears above panels with tabs arranged in a horizontal row; panels stack vertically below; best for most contexts with 3–7 short labels.
- **Vertical:** Tablist appears to the left of panels with tabs arranged in a vertical column; panels appear beside the tablist; works well when labels are longer or available width is limited.

## States

- **Default:** Unselected tabs appear as text labels with no background or border.
- **Selected:** The active tab receives a filled background and border in either primary (red) or secondary (dark teal) brand color with white text.
- **Hover:** Unselected tabs provide visual feedback on pointer hover.
- **Focus:** Keyboard focus shows a visible outline; only the active tab is in the tab order, with arrow keys navigating between tabs.

**Light/dark mode behavior:** Container and panel backgrounds adapt to theme. Selected tab button brand colors remain consistent across modes. In dark mode, <code>page/background</code> resolves to <code>brand/secondary-dark</code> (#01232D) and <code>section/background</code> resolves to <code>brand/secondary</code> (#003442) — note these are distinct dark-mode values that maintain visual contrast between the tablist/panels container and the active panel.

## Accessibility

### Keyboard navigation

Tabs must support full keyboard interaction. The tablist uses a roving tabindex pattern — only the active tab is in the natural tab order.

- **Tab:** Moves focus into and out of the tablist as a single stop.
- **Arrow Right / Arrow Down:** Move focus to the next tab (Right for horizontal tablists, Down for vertical).
- **Arrow Left / Arrow Up:** Move focus to the previous tab (Left for horizontal, Up for vertical).
- **Home:** Move focus to the first tab.
- **End:** Move focus to the last tab.
- **Enter / Space:** Activate the focused tab if not using automatic activation.

### Screen readers

Use semantic ARIA roles to expose the tab structure correctly to assistive technology:

- Use <code>role="tablist"</code> on the tablist container.
- Use <code>role="tab"</code> on each tab button.
- Use <code>role="tabpanel"</code> on each panel.
- Provide an <code>aria-label</code> on the tablist to describe the group of tabs.
- Use descriptive, unique label text on each tab button.
- Ensure panel content is accessible when displayed, including proper heading hierarchy and image alt text.

### ARIA attributes

- **aria-selected:** Set to <code>true</code> on the active tab, <code>false</code> on all others.
- **aria-controls:** Link each tab button to its panel via the panel's ID.
- **aria-labelledby:** Link each panel back to its controlling tab via the tab's ID.
- **aria-label:** Apply to the tablist element to name the tab group (e.g. “Content sections”).
- **tabindex:** Active tab uses <code>0</code>; inactive tabs use <code>-1</code>.

### Focus management

Provide a clear, visible focus indicator on all tab buttons. The Delta Design System uses a 2px outline in <code>brand/primary</code> (#FF5247) with a 2px offset. When a tab is activated via keyboard, consider whether to move focus to the panel or keep it on the tab — both patterns are valid; choose based on context and content length.

### Color contrast

All text and interactive elements must meet WCAG 2.1 AA contrast requirements (4.5:1 for normal text, 3:1 for large text and UI components):

- Default tab text (<code>text/default</code> on transparent) relies on the background behind the tablist.
- Selected tab text (white on <code>brand/primary</code> #FF5247) meets 3:1.
- Selected tab text (white on <code>brand/secondary</code> #003442) exceeds 4.5:1.
- Panel body text (<code>text/default</code> on <code>section/background</code>) must remain compliant in both modes.
- Do not rely on color alone to indicate selected state.

## Usage

### When to use

Use Tabs when you need to:

- Organize related content into distinct, parallel categories.
- Allow users to switch between views without page navigation.
- Reduce visual complexity by hiding secondary content until requested.
- Group settings, configuration options, or comparable items.

### When not to use

Do not use Tabs when:

- Content should be viewed simultaneously.
- Information follows a sequential process.
- Users need to see all content to complete a task.
- There are more than 7–8 tabs.

### Content guidelines

- **Keep labels concise:** 1–2 words per tab label.
- **Use parallel structure:** Keep tab label grammar consistent.
- **Default to the most important tab:** Show commonly needed content first.
- **Ensure panel independence:** Each panel should make sense on its own.
- **Write descriptive labels:** Avoid generic labels like “Tab 1” or “More”.

### Placement

Use the horizontal tablist direction for most contexts, particularly with 3–7 short labels. Use the vertical tablist when labels are longer, available width is limited, or when creating sidebar-style navigation.

### JavaScript

```js
// Tab switching functionality
const tablist = document.querySelector('[role="tablist"]');
const tabs = tablist.querySelectorAll('[role="tab"]');
const panels = document.querySelectorAll('[role="tabpanel"]');

// Handle tab clicks
tabs.forEach(tab => {
  tab.addEventListener('click', () => {
    switchTab(tab);
  });
});

// Handle keyboard navigation
tablist.addEventListener('keydown', (e) => {
  const currentTab = document.activeElement;
  const currentIndex = Array.from(tabs).indexOf(currentTab);

  let nextIndex = currentIndex;

  // Horizontal tablist: Left/Right arrows
  if (tablist.parentElement.dataset.direction === 'horizontal') {
    if (e.key === 'ArrowRight') nextIndex = (currentIndex + 1) % tabs.length;
    if (e.key === 'ArrowLeft') nextIndex = (currentIndex - 1 + tabs.length) % tabs.length;
  }
  // Vertical tablist: Up/Down arrows
  else {
    if (e.key === 'ArrowDown') nextIndex = (currentIndex + 1) % tabs.length;
    if (e.key === 'ArrowUp') nextIndex = (currentIndex - 1 + tabs.length) % tabs.length;
  }

  if (e.key === 'Home') nextIndex = 0;
  if (e.key === 'End') nextIndex = tabs.length - 1;

  if (nextIndex !== currentIndex) {
    e.preventDefault();
    tabs[nextIndex].focus();
    // Auto-activate on arrow key focus (remove for manual activation pattern)
    switchTab(tabs[nextIndex]);
  }
});

function switchTab(newTab) {
  const newPanelId = newTab.getAttribute('aria-controls');
  const newPanel = document.getElementById(newPanelId);

  // Deselect all tabs and hide all panels
  tabs.forEach(tab => {
    tab.setAttribute('aria-selected', 'false');
    tab.setAttribute('tabindex', '-1');
  });

  panels.forEach(panel => {
    panel.setAttribute('hidden', '');
  });

  // Select new tab and show its panel
  newTab.setAttribute('aria-selected', 'true');
  newTab.setAttribute('tabindex', '0');
  newPanel.removeAttribute('hidden');
}
```
