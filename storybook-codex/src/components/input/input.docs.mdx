# Inputs

Inputs collect, validate, and communicate form data across text entry, selections, and toggles.

## Anatomy

- Label/legend: context for the field value
- Control surface: border, padding, and state color
- Value/placeholder: content text inside text-like controls
- Helper message: optional supporting guidance
- Accessory icon: optional leading/trailing indicator

## States

- Text/select/password: `default`, `value`, `disabled`, `invalid`, `read-only`, `valid` (plus `open`/`show` where applicable)
- Checkbox/radio: `default`, `disabled`, `invalid`, `valid`
- Switch: `default`, `disabled` with `checked/unchecked` and direction support

## Accessibility

- Use semantic form controls (`input`, `select`, `fieldset`, `legend`, `label`).
- Keep clear text labels and link helper text via `aria-describedby` when needed.
- Use `aria-invalid="true"` when validation fails.
- Preserve keyboard support for checkbox/radio/switch interactions.
- Maintain minimum 44x44px clickable target in production layouts.

## Usage

### Do

- Use state colors consistently to communicate validation status.
- Pair labels and helper text with clear, actionable copy.
- Keep option list ordering predictable and stable.

### Don’t

- Do not use placeholder text as the only label.
- Do not rely only on color to communicate errors.
- Do not mix multiple validation tones in one field unless intentional.
