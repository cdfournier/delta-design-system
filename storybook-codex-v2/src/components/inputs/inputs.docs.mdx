# Inputs

Form inputs allow users to enter information, make selections, and provide data. The Delta Design System provides a comprehensive set of input components that are accessible, consistent, and easy to implement.

## Anatomy

Input components share common structural elements:

- **Input field:** The interactive area where users enter or select data.
- **Label:** Text that describes what information is expected.
- **Border:** Visual container that defines the input boundary.
- **Helper text (optional):** Additional guidance or instructions.
- **Validation message (optional):** Feedback indicating valid, invalid, or error states.
- **Icon (optional):** Visual indicator for input type, state, or action.

## Variants

The Delta Design System includes multiple input types, each designed for specific data entry needs.

### Text input

Single-line text fields for entering short text, email addresses, phone numbers, URLs, and numerical data. Text inputs are the most common input type.

<Canvas of={ComponentStories.VariantTextFigure} />

### Select

Dropdown menus that allow users to choose a single option from a predefined list. Use selects when there are 5 or more options.

<Canvas of={ComponentStories.VariantSelectFigure} />

### Checkbox

Square controls for selecting multiple options from a list or toggling a single option on or off.

<Canvas of={ComponentStories.VariantCheckboxFigure} />

### Radio

Circular controls for selecting exactly one option from a group. Radio buttons should always appear in groups of two or more.

<Canvas of={ComponentStories.VariantRadioFigure} />

### Switch

Toggle controls for turning a setting on or off. Switches provide immediate feedback and are best for settings that take effect instantly.

<Canvas of={ComponentStories.VariantSwitchFigure} />

### Password

Secure text fields that obscure entered characters with bullets or asterisks. Typically include a show/hide toggle for visibility control.

<Canvas of={ComponentStories.VariantPasswordFigure} />

### Messages

Contextual text that provides helper information, validation feedback, or error messages. Messages appear below input fields and use color coding to indicate state.

<Canvas of={ComponentStories.VariantMessagesFigure} />

## States

All input components share common interaction states that provide visual feedback to users.

### Default

The resting state of an input before any user interaction. Borders use `global/gray-medium` (`#CCCCCC`).

<Canvas of={ComponentStories.StateDefaultFigure} />

### Hover

The cursor-over state. No visual border change occurs on hover. The cursor changes to indicate the field is interactive.

### Focus

Applied when an input receives keyboard or pointer focus. The browser default focus outline is used. Never remove focus indicators.

### Disabled

Indicates the input is not currently interactive. Uses `status/disabled` (`#808080`) for both border and text color.

<Canvas of={ComponentStories.StateDisabledFigure} />

### Read-only

Displays data that cannot be edited but can be read and copied. Uses `status/read-only` (`#333333`) for border differentiation.

<Canvas of={ComponentStories.StateReadOnlyFigure} />

### Valid

Indicates successful validation. Uses `status/valid` (`#007000`) for border and validation message.

<Canvas of={ComponentStories.StateValidFigure} />

### Invalid

Indicates validation failure. Uses `status/invalid` (`#C00C00`) for border and error message.

<Canvas of={ComponentStories.StateInvalidFigure} />

**Light/Dark mode behavior:** Background and text colors invert between light and dark modes for readability. Status colors and brand colors are mode-invariant.

## Accessibility

### Keyboard navigation

All input components must be fully keyboard accessible:

- **Tab:** Moves focus to the next input or interactive element.
- **Shift + Tab:** Moves focus to the previous input or interactive element.
- **Space:** Toggles checkboxes, radio buttons, and switches.
- **Enter:** Submits forms when appropriate.
- **Arrow keys:** Navigate between radio options in a group.

### Screen readers

Use semantic HTML and proper labeling. Every input must have an associated label. Group related options with `<fieldset>` and `<legend>`.

### ARIA attributes

- `aria-label` when no visible label is present.
- `aria-labelledby` to associate the input with label text.
- `aria-describedby` to connect helper/validation messages.
- `aria-invalid="true"` when validation fails.
- `aria-required="true"` for required fields.
- `role="switch"` for switch controls.

### Focus management

Inputs rely on browser default focus indicators. Never remove focus indicators.

### Touch targets

Maintain minimum touch target size of at least `44px` where possible.

### Color contrast

Ensure border/text states meet WCAG contrast requirements in both light and dark modes.

### Error prevention and recovery

Provide descriptive validation feedback and clear recovery actions.

## Usage

### When to use

Use input components when you need to:

- Collect user-entered data in forms.
- Let users choose one or more options.
- Capture account credentials and secure values.
- Provide settings toggles.
- Display validation feedback close to the field.

### When not to use

Do not use input components when:

- You only need navigation (use links/buttons).
- Content is static and non-editable.
- A simpler control communicates intent more clearly.

### Content guidelines

- Write clear, concise labels.
- Use placeholder text for examples, not as labels.
- Write actionable error messages.
- Keep helper text brief.
- Use sentence case.

### Placement

Stack inputs vertically for most forms. Place labels consistently above fields. Use multi-column layouts only for short, related groups.

## Code reference

### HTML

#### Text input

```html
<!-- Basic text input -->
<label for="username">Username</label>
<input type="text" id="username" name="username" />

<!-- With helper text -->
<label for="email">Email address</label>
<input type="email" id="email" name="email" aria-describedby="email-help" />
<p id="email-help">We'll never share your email.</p>

<!-- With validation -->
<label for="email-invalid">Email address</label>
<input type="email" id="email-invalid" name="email"
       aria-invalid="true" aria-describedby="email-error" />
<p id="email-error" role="alert">Please enter a valid email address.</p>
```

```html
<label for="country">Country</label>
<select id="country" name="country">
  <option value="">Select a country</option>
  <option value="us">United States</option>

  <option value="uk">United Kingdom</option>
```

### JavaScript

Example validation script for form inputs:

```javascript
// Email validation example
const emailInput = document.getElementById('email');
const emailError = document.getElementById('email-error');

emailInput.addEventListener('blur', () => {
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  const isValid = emailRegex.test(emailInput.value);

  emailInput.setAttribute('aria-invalid', !isValid);

  if (isValid) {
    emailError.textContent = '';
    emailError.style.display = 'none';
  } else {
    emailError.textContent = 'Please enter a valid email address.';
    emailError.style.display = 'block';
  }
});

// Password visibility toggle
const passwordInput = document.getElementById('password');
const toggleButton = document.getElementById('toggle-password');

toggleButton.addEventListener('click', () => {
  const type = passwordInput.type === 'password' ? 'text' : 'password';
  passwordInput.type = type;
  toggleButton.textContent = type === 'password' ? 'Show' : 'Hide';
  toggleButton.setAttribute('aria-label',
    type === 'password' ? 'Show password' : 'Hide password'
  );
});
```
